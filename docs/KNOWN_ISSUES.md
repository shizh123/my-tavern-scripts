# 已知问题 (Known Issues)

## 架构问题：Prompt Injection 中的数据修改可能导致楼层数据错乱

### 问题描述

项目中存在架构设计问题：Prompt Injection 系统在多处使用 `message_id: -1` 修改游戏数据，这可能导致数据被写入错误的楼层。

### 根本原因

- **Prompt Injection** 在 `CHAT_COMPLETION_PROMPT_READY` 事件中运行（AI生成**之前**）
- 此时 `message_id: -1` 指向"用户当前正在查看的楼层"
- 如果用户正在查看旧楼层（如返回查看历史记录），数据会被错误地写入旧楼层
- **游戏逻辑系统** 在 `MESSAGE_RECEIVED` 事件中运行（AI生成**之后**），使用明确的 `targetMessageId`

### 正确的架构模式（参考秦璐项目）

- ✅ **Prompt Injection**：只读取数据（`getMvuData`），用于生成提示词，**不修改数据**
- ✅ **游戏逻辑系统**：所有数据修改都在此完成，使用明确的 `message_id`

**注意**：秦璐项目虽然遵循了这个架构分离，但仍存在类似问题：

- 经常出现时间没有刷新，需要 ROLL 才能生效
- 或者需要重新加载脚本才能看到正确的数据
- 这说明即使架构分离正确，仍然存在数据同步/刷新的时序问题

### 更深层的架构问题：事件时序和数据同步

即使将数据修改移至游戏逻辑系统，仍可能存在以下问题：

#### 问题1：事件执行顺序不确定
- `MESSAGE_RECEIVED` 和 `GENERATION_ENDED` 事件的触发顺序可能不稳定
- 两个事件都使用 `setTimeout(..., 300)` 延迟执行，可能存在竞态条件
- 如果数据修改和界面刷新的顺序错误，会导致界面显示旧数据

#### 问题2：MVU 数据绑定的刷新时机
- 状态栏使用 `message_id: -1` 读取"当前查看楼层"的数据
- 但数据修改可能发生在界面组件读取数据之前或之后
- Vue 的响应式系统可能无法及时捕获 MVU 数据的变化

#### 问题3：多楼层数据隔离不完善
- 每个楼层应该有独立的数据快照
- 但如果事件处理时机不对，可能读取到其他楼层的数据
- ROLL 操作会清除事件处理记录，重新执行逻辑，所以 ROLL 后可能"修复"问题

#### 可能的解决方向

1. **同步数据修改和界面刷新**
   - 数据修改后立即广播 `IFRAME_DATA_REFRESH` 事件
   - 确保界面组件在数据修改完成后才读取新数据

2. **改进事件去重机制**
   - 当前使用 `processedEvents` Set 去重
   - 可能需要更精确的事件处理状态追踪

3. **强制刷新机制**
   - 在关键数据变更后，强制 Vue 组件重新读取 MVU 数据
   - 或者使用更可靠的数据绑定方式

4. **调试工具**
   - 添加详细的事件触发和数据修改日志
   - 追踪每个楼层的数据变化历史

4. Prompt Injection 会将数据写入用户当前查看的旧楼层，而不是新生成的楼层

### 平衡性 #1: 数值增长过快（依存度/部位进度） ✅ 已修复

**发现日期**: 2026-01-15
**修复日期**: 2026-01-16

**问题描述**:

- 测试发现：2-3天就能把全部数值做满
- Day 1 只进入场景5，出来后依存值可以达到 70 以上
- 即使玩家什么都没做，数值也会自动增长
- 目标应该是：4天梦境中把全部数值做到80-100%

**根本原因分析**:

1. **AI回复检测导致被动增长**：
   - 部位开发检测同时检测玩家输入和AI回复
   - AI描写中经常出现"胸部"、"下体"等关键词
   - 导致即使玩家不主动做什么，数值也会增长

2. **场景5错误触发**：
   - 场景5是剧情场景，但也被标记为"梦境"阶段
   - 触发了通用的部位开发检测

3. **每晚上限过高**：
   - 原设计每晚25%上限，3-4晚就能做满

**修复方案**:

1. **只检测玩家输入**（核心修复）：
   - 移除对AI回复的关键词检测
   - 玩家不主动做 → 数值不涨

2. **排除场景5**：
   - 场景5有自己的"完成度"系统，不触发部位开发

3. **降低每晚上限**：
   - 从25%降到20%
   - 4晚 × 20% = 80%，专注开发可达100%

```typescript
// index.ts - 只检测玩家输入，排除场景5
const isInScene5 = data.梦境数据.场景5?.已进入 === true;
if (data.世界.游戏阶段 === '梦境' && !isInScene5) {
  // 只检测玩家输入的关键词（玩家不主动做，数值不涨）
  const userInputProgress = userText ? detectBodyPartProgress(userText) : {...};
  // ...
}

// dreamKeywordDetection.ts - 降低每晚上限
const NIGHTLY_PROGRESS_CAP = 20; // 从25降到20
```

**修改的文件**:

- `src/赵霞/脚本/游戏逻辑/index.ts:842-857` - 只检测玩家输入，排除场景5
- `src/赵霞/脚本/游戏逻辑/dreamKeywordDetection.ts:161-167` - 每晚上限从25%降到20%

**设计说明**:

- 依存度 = 所有部位进度的平均值
- 道德底线 = 100 - 依存度
- 境界 = 依存度 / 20（向上取整）
- 4天梦境中，通过主动互动，全部数值应达到80-100%

**优先级**: 🔥 高（严重影响游戏平衡性和可玩性）

### Bug #2: 状态栏初始化时 ZodError 报错

**发现日期**: 2026-01-15

**问题描述**:
- 角色卡首次加载或刷新时，浏览器控制台出现大量 ZodError 错误
- 错误信息：所有主要字段（世界、赵霞状态、梦境数据、现实数据、结局数据）都是 `undefined`
- 错误来源：`src/util/mvu.ts:20` - `schema.parse()` 在数据为空时抛出错误

**问题原因**:
1. **数据未初始化**: 角色卡刚加载时，MVU 变量 `stat_data` 可能还未创建
2. **强制解析**: `defineMvuDataStore` 使用 `schema.parse({})` 强制解析空对象
3. **缺少默认值处理**: 没有使用 `safeParse` 并提供默认值的机制

**实际影响**:
- ⚠️ 控制台错误信息干扰调试
- ⚠️ `errorCatched` 可能会在 SillyTavern 中显示错误通知
- ✅ 不影响游戏正常运行（错误被捕获后，后续轮询会修复数据）

**已修复**:
修改了 `src/util/mvu.ts:20-24`，使用 `safeParse` 避免初始化错误：

```typescript
// 修复前
const data = ref(schema.parse(_.get(getVariables(variable_option), 'stat_data', {}))) as Ref<z.infer<T>>;

// 修复后
const stat_data = _.get(getVariables(variable_option), 'stat_data', {});
const parseResult = schema.safeParse(stat_data);
const initialData = parseResult.success ? parseResult.data : schema.parse({});
const data = ref(initialData) as Ref<z.infer<T>>;
```

### Bug #18: 场景1-4在10:00时没有触发退出描写 ✅ 已修复

**发现日期**: 2026-01-16
**修复日期**: 2026-01-16

**问题描述**:
- 场景1-4梦境进行到第9或10轮时（接近10:00）
- 游戏时间到达10:00，游戏阶段正确切换为"日常"
- **但是没有出梦描写**（AI没有生成醒来场景的文本）

**根本原因**:
时序问题（与 Bug #13 完全相同）：
1. `promptInjection.ts` 在 AI 生成**之前**执行（CHAT_COMPLETION_PROMPT_READY）
2. 时间推进在 AI 生成**之后**执行（GENERATION_ENDED）
3. 当时间是 09:00 时，`checkDreamExit` 检测 `当前小时 === 10` 返回 false
4. AI 生成普通梦境内容（而不是出梦描写）
5. 时间推进到 10:00
6. `index.ts` 检测到 10:00，切换游戏阶段
7. **问题**：AI 已经生成了梦境内容，没有出梦描写！

**修复方案**:
将退出检测时间从 `=== 10` 改为 `>= 9`，让 AI 在 09:00 时就生成出梦描写：

```typescript
// promptInjection.ts - checkDreamExit
// 修复前
return TimeSystem.isWakeUpTime(data); // 检查 === 10

// 修复后
return data.世界.当前小时 >= 9; // 09:00 时触发
```

**修复后流程**:
- 09:00 时触发退出检测 → AI 生成出梦描写 → 时间推进到 10:00 → 游戏阶段切换为日常

**修改的文件**:
- `src/赵霞/脚本/游戏逻辑/promptInjection.ts:1826-1852` - 退出检测改为 09:00
- `src/赵霞/脚本/游戏逻辑/promptInjection.ts:1747-1749` - 移除"时间已到10:00"文本
- `src/赵霞/脚本/游戏逻辑/index.ts:285-287` - 更新注释说明

**注意**:
修复后场景1-4的实际可用回合从约11回合变为约10回合：
- 22:00 进入 → 23:00, 00:00, ..., 08:00 梦境内容 → 09:00 出梦描写

**优先级**: 🔥 高（影响玩家体验和剧情连贯性）

### 二、记忆连贯性系统（替代记忆混乱度）

#### 2.1 核心概念

**废弃**: `记忆混乱度`（负向惩罚，机制不透明，玩家体验差）
**新增**: `记忆连贯性`（正向奖励 + 关键判定条件）

```typescript
// 记忆连贯性等级
type 记忆连贯性 = 0 | 1 | 2 | 3;

// 计算规则（进入场景5时计算）
let 连贯性 = 0;
if (已完成场景.includes(1)) 连贯性 = 1;
if (已完成场景.includes(1) && 已完成场景.includes(2)) 连贯性 = 2;
if (已完成场景.includes(1) && 已完成场景.includes(2) && 已完成场景.includes(3)) 连贯性 = 3;
// 注意：必须按顺序完成才计入连贯性
// 场景4不计入（时间线在场景5之后）
```

#### 2.2 玩家路线分析

**路线A：速攻路线（大多数玩家）**
```
Day 1 白天: 进入场景5（连贯性0）
结果: 能完成12步，但无前置记忆加成，无法触发完美真爱结局
```

**路线B：平衡路线**
```
Day 1 晚: 场景1
Day 2 白天: 进入场景5（连贯性1）
Day 2 晚: 场景2
Day 3 白天: 继续场景5（连贯性仍为1）
结果: 有部分记忆，体验较好
```

**路线C：完美记忆路线（隐藏最优解）**
```
Day 1 晚22:00: 场景1 → Day 2早10:00退出
Day 2 晚22:00: 场景2 → Day 3早10:00退出
Day 3 晚22:00: 场景3 → Day 4早10:00退出
Day 4 白天10:00: 进入场景5（连贯性3！）
Day 4 白天: 推进8步 → 19:00退出
Day 4 晚22:00: 可进入场景4（可选）
Day 5 白天10:00: 继续场景5，完成剩余4步 + 自由剧情
Day 5, 20:00: 结算
结果: 完美记忆连贯，可触发完美真爱结局！
```

#### 3.4 记忆混乱的新定位

**不是惩罚机制，是结局指示器**：
- 记忆混乱增加 = 提示玩家"这不是完美路线"
- 不影响游戏进程，不阻止玩家任何操作
- 玩家第一次游玩可能不明白意义，第二次就会懂
- 记忆混乱 > 0 = 无法达成完美真爱结局

**触发规则**：
- 记忆连贯性 = 3 → 记忆混乱**不增加**
- 记忆连贯性 < 3 → 场景5每完成一步，记忆混乱 +5

### 四、摘要记忆机制（核心机制）

- 场景5进入时，读取已完成场景的摘要，注入到AI提示词

#### 4.2 摘要记忆的作用

#### 4.3 技术要求


### 五、结局系统重设计

#### 5.1 结局类型

| 结局 | 触发条件 | 描述 |
|-----|---------|-----|
| **完美真爱结局** | 记忆连贯性=3 + 场景5完成度≥100% + 依存度≥80 | 赵霞从记忆深处就属于玩家，真正的灵魂伴侣 |
| **真好结局（堕落）** | 场景5完成度≥80% + 全场景正确重构 | 赵霞精神被重塑，完全堕落属于玩家 |
| **假好结局** | 部分场景正确重构 | 表面幸福，内心仍有挣扎 |
| **普通结局** | 未完成关键条件 | 循环继续/重置 |
| **坏结局** | 丈夫怀疑度溢出 | 被发现，关系破裂 |

#### 5.2 完美真爱结局的特殊性

**触发条件**（全部满足）:
1. 记忆连贯性 = 3（完成场景1-2-3后再进入场景5）
2. 场景5完成度 ≥ 100%
3. 依存度 ≥ 80
4. 道德底线 ≤ 30

**结局内容差异**:
- 赵霞不是被"洗脑"，而是从记忆深处就认定玩家
- 她主动摘下戒指，说出"从一开始就应该是你"
- 结局描写更偏向"命中注定"而非"精神控制"
- 后日谈内容不同（真正的幸福 vs 堕落的满足）

#### 5.3 记忆连贯性与记忆混乱的关系

**新规则**:
- 记忆连贯性 = 3 时，**不会增加记忆混乱**
- 记忆连贯性 < 3 时，场景5中某些操作会增加记忆混乱
- 记忆混乱过高会阻止触发好结局

**设计意图**:
- 完美记忆路线 = 记忆连贯 = 不会混乱
- 速攻路线 = 记忆不连贯 = 有混乱风险
- 给玩家一个"正确顺序"的激励
