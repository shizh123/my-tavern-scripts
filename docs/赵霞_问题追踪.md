### M2. 梦境结束时部位选择验证逻辑缺陷
- **状态**: ✅ 已解决
- **位置**: `src/赵霞/脚本/游戏逻辑/index.ts:207-235`
- **问题描述**:
  - 当 `selectedParts.length === 0` 时，场景直接标记为"已完成"
  - 未区分"未进入梦境"和"进入但未选择"两种情况
- **影响**: 影响结局判定准确性
- **解决方案**: 使用 Schema 中已有的 `已进入` 字段进行状态区分
- **修复内容**:
  - 新增 `hasEntered` 变量读取场景的 `已进入` 状态
  - 三种情况分别处理：
    1. `!hasEntered`: 未进入梦境 → 保持未完成状态，不做处理
    2. `hasEntered && selectedParts.length > 0`: 进入且选择 → 正常判定
    3. `hasEntered && selectedParts.length === 0`: 进入但未选择 → 标记为错误
- **解决日期**: 2026-01-15

## 🏗️ 系统架构设计（2026-01-14）

### 核心原则：两套独立系统

Prompt注入系统应该分为两个独立的子系统，各司其职：

#### 系统A：事件触发系统（场景切换类）

**特点**：条件满足时一次性触发，替换用户消息

| 事件类型 | 触发条件 | 作用 |
|----------|----------|------|
| 梦境入口 | 23:00-01:00 + 睡眠关键词 | 替换消息，切换到梦境阶段 |
| 梦境退出 | 07:00/10:00自动 | 替换消息，切换回日常阶段 |
| 场景5入口 | 白天 + 安眠药关键词 | 替换消息，进入特殊梦境 |
| 场景5退出 | 20:00强制 | 替换消息，结束特殊梦境 |
| 打断事件 | 苏文回家等条件 | 替换消息，触发打断剧情 |
| 结局触发 | 各结局条件达成 | 替换消息，进入结局 |
| 纯爱错误路线 | 部位开发达100% | 替换消息，触发警告场景 |

**共同点**：都是"检测条件 → 替换消息 → 改变游戏状态"

#### 系统B：AI行为引导系统（持续注入类）

**特点**：每次对话都注入system提示，影响AI行为

| 阶段 | 注入内容 | 包含信息 |
|------|----------|----------|
| 日常 | 日常引导提示 | 苏文状态、时间环境、互动范围、境界约束 |
| 梦境 | 梦境引导提示 | D9时间段引导、记忆年龄、部位进度、梦境规则 |

**共同点**：都是"根据当前状态 → 生成提示 → 影响AI描写"

#### 两系统关系

```
玩家发消息
    ↓
┌─ 系统A：检测事件触发条件
│  ├─ 触发？→ 替换消息 + 改变状态 + 跳过系统B
│  └─ 未触发？→ 继续
│
└─ 系统B：生成行为引导
   ├─ 日常阶段？→ generateDailyGuidancePrompt()
   └─ 梦境阶段？→ generateDreamGuidancePrompt()
```

**关键设计**：
- 事件触发时（系统A），不需要额外注入行为引导（系统B）
- 替换消息本身已包含足够的AI指令
- 避免重复/冲突的提示信息

#### 待优化项

当前代码问题：
1. `generateStateConsistencyPrompt()` 在梦境中也注入苏文现实状态（不合理）
2. 事件触发时仍然注入一致性提示（冗余）
3. D9引导和状态一致性是分开注入的，应该融合成阶段专用提示

优化目标：
- 日常阶段：`generateDailyGuidancePrompt()` 一个函数搞定
- 梦境阶段：`generateDreamGuidancePrompt()` 一个函数搞定（包含D9+梦境状态）
- 事件触发时：只用替换消息，不额外注入

### 场景1初始化系统（2026-01-14 新增）

## ❓ 常见困惑澄清（给新会话的自己）

### Q1: 场景编号是怎么决定的？

**错误理解**：有安眠药就进场景5

**正确理解**：
- 场景1-4：按天数对应（Day 1=场景1, Day 2=场景2...）
- 场景5：是**额外隐藏关卡**，通过安眠药关键词单独触发
- 两者是**完全独立的入口**，不是同一个判断逻辑

### Q2: 梦境入口在哪里触发？

**错误理解**：界面有"进入梦境"按钮

**正确理解**：
- 游戏伪装成恋爱游戏，**玩家不知道梦境存在**
- 只有当玩家**偶然**在22:00-01:00输入睡眠相关内容，才会触发梦境
- 触发机制在脚本层（promptInjection），不是界面按钮
- 场景5同理：玩家在08:00-19:00输入安眠药相关内容触发

### Q3: 部位选择有什么用？

**错误理解**：选择后只能开发选中的部位

**正确理解**：
- 选择后，玩家可以**自由开发任意部位**
- 选择的作用是**结算判定**：只有精准选择正确部位才算"正确重构"
- 多选会导致不算正确重构（贪心惩罚）

### Q4: 状态栏的"本次目标"是什么？

**正确理解**：
- 这是对玩家的**暗示**，告诉玩家正确部位是什么
- 例如"将初吻对象替换为你" → 暗示选择"嘴巴"
- 但玩家可能贪心多选，导致结算失败

### Q5: UI的 currentSceneNumber 逻辑是否正确？

**当前代码问题**：

```typescript
if (realData.value.拥有安眠药) {
  return 5;  // ← 错误：只要有安眠药就显示场景5
}
return Math.min(day, 4);
```

**应该修改为**：从 Schema 读取实际进入的场景编号，而不是预测

### Q6: 场景可以补做吗？

**错误理解**：错过的场景可以在后面补

**正确理解**：

- 场景**绑定日期**：Day 1=场景1, Day 2=场景2, Day 3=场景3, Day 4=场景4
- **错过即永久错过**：Day 2无法再进入场景1
- 结算日会判定玩家错过了哪些场景，影响最终结局
- 这增加了游戏的紧迫感和策略性

### Q7: 场景5什么时候可以进入？

**正确理解**：

- 场景5是**额外隐藏关卡**，不受日期绑定
- 只要有安眠药 + 在08:00-19:00输入相关关键词即可触发
- 可以在任意天数（Day 1-5）进入
- 但场景5需要一定的部位开发基础才能获得好效果

## 🚨 坏结局系统（2026-01-14 实现）

### 系统归属

**系统A：事件触发系统**
- 条件满足时一次性触发
- 替换用户消息
- 触发后跳过系统B

### 触发条件

| 结局类型 | 触发条件 | 状态 |
|---------|---------|------|
| 发现结局 | `丈夫怀疑度 >= 100` | ✅ 已实现 |
| 精神崩溃结局 | `记忆混乱度 >= 100` | ⏳ 基础实现 |

### 核心机制

1. **ROLL防护**：一旦触发坏结局，游戏状态会被锁定
   - 即使玩家ROLL消息使怀疑度回到100以下

#### 📌 剧情锚点机制详解

**核心原理**：每个阶段定义若干**锚点事件(Anchor Events)**，AI回复后检测是否包含锚点关键词，累计满足条件后自动推进到下一阶段。

**1. 锚点事件定义**

```typescript
// 每个阶段的锚点事件配置示例
{
  phase: 1,
  name: '午餐',
  anchorEvents: ['苏文入座', '敬酒', '苏文饮酒'],  // 必须发生的事件
  minTurns: 2,  // 最少对话轮数
}
```

**2. 锚点检测流程**

```
玩家发送消息
    ↓
AI生成回复
    ↓
[index.ts] MESSAGE_RECEIVED事件触发
    ↓
processTurnEnd(state, aiText) 检测锚点
    ↓
detectCompletedAnchors(aiResponse, phase) 关键词匹配
    ↓
更新 completedAnchors 数组
    ↓
shouldAdvancePhase() 检查是否满足推进条件
    ↓
满足 → advanceToNextPhase() 推进
不满足 → 继续当前阶段
```

**3. 锚点检测算法**

```typescript
// 关键词可以是字符串或正则表达式
const ANCHOR_KEYWORDS: Record<string, (string | RegExp)[]> = {
  '纪念日提及': ['结婚纪念', '纪念日', '周年'],
  '苏文入座': ['苏文.*坐', '入座', '落座', '坐下'],
  '敬酒': ['举杯', '干杯', '敬酒', '碰杯'],
  // ...
};

function detectCompletedAnchors(aiResponse: string, phase: number): string[] {
  const phaseConfig = TRUE_ENDING_PHASES[phase];
  const completed: string[] = [];

  for (const anchor of phaseConfig.anchorEvents) {
    const keywords = ANCHOR_KEYWORDS[anchor];
    for (const kw of keywords) {
      if (typeof kw === 'string') {
        if (aiResponse.includes(kw)) completed.push(anchor);
      } else {
        if (kw.test(aiResponse)) completed.push(anchor);
      }
    }
  }
  return completed;
}
```

**4. 阶段推进条件**

```typescript
function shouldAdvancePhase(state: TrueEndingState): boolean {
  const phaseConfig = TRUE_ENDING_PHASES[state.currentPhase];

  // 条件1：对话轮数满足最小要求
  if (state.turnsInPhase < phaseConfig.minTurns) return false;

  // 条件2：所有锚点事件都已完成
  const allAnchorsComplete = phaseConfig.anchorEvents.every(
    anchor => state.completedAnchors.includes(anchor)
  );

  return allAnchorsComplete;
}
```

// 距离结局结算的小时数（Day 5, 07:00 结算）
const hoursUntilEnding = computed(() => {
  const currentDay = worldData.value.当前天数;
  const currentHour = worldData.value.当前小时;

## 🔄 普通结局系统（2026-01-15 实现）

### 设计背景

在原设计中，未完成全部场景的玩家会触发"坏结局"。但这不合理：
- 玩家可能从未发现梦境入口
- 玩家可能进入了部分梦境但未完成全部5个场景
- 这些情况不应该等同于"被发现"或"精神崩溃"

因此新增**普通结局**，表示"时间循环重置"。

### 触发条件

```
普通结局触发条件：
1. 时间到达 Day 5, 12:00（结局判定时间）
2. 未触发坏结局（怀疑度<100 且 混乱度<100）
3. 未完成全部5个梦境场景
```

### 四种结局完整定义

| 结局类型 | 触发条件 | 描述 |
|----------|----------|------|
| **真好结局** | 完成全部5个场景 + 全部正确选择 | 记忆重构成功，打破循环 |
| **假好结局** | 完成全部5个场景 + 有错误选择 | 记忆重构不完美，仍能打破循环 |
| **坏结局** | 怀疑度≥100 或 混乱度≥100 | 被发现/精神崩溃 |
| **普通结局** | 时间到 + 无坏结局 + 未完成全部场景 | 时间循环重置 |
