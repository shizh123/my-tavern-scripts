# 秦璐角色卡 - 功能文档

## 脚本架构

### 游戏逻辑脚本 (`脚本/游戏逻辑/`)

**重构说明**：原先分散在多个脚本中的游戏逻辑已合并为单一脚本，消除了数据竞争问题。

监听三种事件触发游戏逻辑处理（均延迟 300ms）：

| 事件 | 作用 | 去重处理 |
|------|------|----------|
| `MESSAGE_RECEIVED` | 新消息接收 | 同一 message_id 只处理一次 |
| `MESSAGE_SWIPED` | 消息 ROLL | 清除旧记录后重新处理 |
| `GENERATION_ENDED` | 生成结束（备选） | 使用 `getLastMessageId()` 获取实际 ID |

按以下顺序**同步执行**（只读写一次变量）：

| 步骤 | 模块 | 功能 |
|------|------|------|
| 1 | timeAdvance.ts | 时间+1小时、位置追踪、疑心值、巡逻检测、念头进度 |
| 2 | thoughtCheck.ts | 念头成熟/过期处理、数值变化、阶段提升 |
| 3 | impression.ts | 根据疑心值更新印象文字 |
| 4 | appearance.ts | 检查外观约束、强制调整不满足要求的外观 |
| 5 | logCleanup.ts | 清理念头植入日志（保留3楼内的日志） |

### 其他独立脚本

| 脚本 | 触发时机 | 功能 |
|------|----------|------|
| 习惯注入 | CHAT_COMPLETION_PROMPT_READY | 注入状态一致性提示、习惯列表、念头植入通知、强制事件（打断/巡逻） |
| 核心数值保护 | COMMAND_PARSED | 拦截 AI 直接修改核心数值 |
| 变量结构 | 加载时 | 注册 MVU Schema、初始化 fusion_hint 变量 |

## 脚本功能详解

### 1. 时间推进模块 (`脚本/游戏逻辑/timeAdvance.ts`)

#### 时间系统

- 每轮对话自动推进 1 小时
- 跨日自动更新日期（星期几同步更新）
- 初始化后首次消息跳过时间推进

#### 时间跳跃检测

检测玩家输入中的时间跳跃关键词，支持以下类型：

| 类型 | 关键词示例 | 跳跃效果 |
|------|-----------|---------|
| 天数跳跃 | 一周后、一个月后 | +7天、+30天 |
| 小时跳跃 | 几小时后 | +3小时 |
| 时间段跳跃 | 早上、下午、晚上 | 跳到7点、15点、19点 |
| 场景跳跃 | 吃早饭、吃晚饭 | 跳到8点、19点 |

#### 位置追踪

直接从 `世界.地点` 变量读取当前位置（AI 通过 MVU 命令更新，比从 AI 文本匹配更可靠）：

- 脚本解析 `世界.地点` 字段，如 `家 - 主角房间`
- 判断是否包含"主角房间"来确定危险状态
- 位置变化时自动更新 `系统追踪.当前所在位置` 和 `进入当前位置时间`

#### 苏文作息系统

> **星期显示**：界面会根据日期自动显示星期几，并用颜色区分：
> - 🟡 金色标签：上午班工作日（周一、三、五），苏文 8:00-12:00 上班
> - 🔵 蓝色标签：下午班工作日（周二、四），苏文 13:00-18:00 上班
> - 🔴 红色标签：周末，苏文全天在家（高危）

- 13:00-18:00：在家（客厅）
- 18:00-19:00：在家（餐厅/厨房）
- 19:00-22:00：在家（客厅）

**工作日 - 下午班 (周二、周四)**：

#### 疑心值系统

**增加条件**（需同时满足）：

- 角色在主角房间
- 苏文在家且清醒（非睡眠/上班/出差）
- 苏文不在同一空间
- 疑心值未被冻结

**增加规则**：

- 危险时间累计 ≥2 小时：疑心值 +5
- 危险时间累计 ≥4 小时：疑心值 +10，触发苏文打断事件

**衰减规则**：

- 每天自然衰减 -2
- 与苏文同处一室时每小时 -1

**冻结机制**：

- 角色向苏文说谎并告知时间时可冻结疑心值
- 超过承诺时间仍在主角房间：疑心值 +8，解除冻结

#### 苏文巡逻系统

**触发条件**：

- 角色在主角房间且处于危险状态
- 没有正在进行的巡逻事件
- 疑心值未冻结
- 距上次巡逻超过 2 小时

**触发概率**：

- 基础概率：15%
- 时间修正：上午×1.0、下午×1.2、晚上×0.8、深夜×0.3
- 疑心值修正：0-20×0.8、20-40×1.0、40-60×1.3、60-80×1.6、80-100×2.0
- 最高概率上限：80%

**触发后果**：

- 苏文设置1-2小时的离开期限
- 超过期限仍在主角房间将触发苏文打断事件

#### 念头开发进度

根据用户输入和 AI 输出与念头内容的相关性增加进度：

- 完全匹配或 ≥5 个关键词匹配：+1.5 小时
- ≥3 个关键词匹配：+1 小时
- ≥1 个关键词匹配：+0.5 小时
- 无匹配：+0 小时

**时间跳跃特殊处理**：

当发生时间跳跃时（如"一周后"、"第二天"等），念头不会因跳过过期时间而消失：

- 念头开发进度 +1 小时（固定值）
- 过期时间延长至新时间 +12 小时

#### 双角色独立疑心值系统

秦璐和苏梦各自维护独立的疑心值和危险时间累计：

| 字段 | 秦璐 | 苏梦 |
|------|------|------|
| 疑心值 | 对秦璐疑心值 | 对苏梦疑心值 |
| 危险时间累计 | 秦璐危险时间累计 | 苏梦危险时间累计 |
| 疑心值冻结 | 对秦璐疑心值冻结 | 对苏梦疑心值冻结 |

#### 强制事件日志系统

脚本会将触发的强制事件（阶段突破、苏文打断、苏文巡逻）写入 `系统追踪.强制事件日志`，支持 ROLL 后重新触发：

```typescript
tracking.强制事件日志.push({
  事件类型: '苏文打断' | '苏文巡逻' | '阶段突破',
  角色: '秦璐' | '苏梦',
  触发楼层: currentFloor,
  触发时间: `${newDate} ${newTime}`,
  已通知AI: false,
  // 打断事件额外字段
  累计时间?: number,
  打断原因?: string,
  // 阶段突破额外字段
  阶段号?: number,
});
```

#### 打断事件借口结果处理

当 AI 判定借口成功/失败后，脚本自动处理后续逻辑：

- **借口成功**：
  - 自动设置疑心值冻结（2 小时）
  - 疑心值不再累积

- **借口失败**：
  - 额外疑心值 +10（总共 +20）
  - 角色必须离开主角房间

#### 安眠药机制（秘籍系统）

安眠药是游戏中唯一的"秘籍"机制，允许玩家临时解锁下一阶段的念头植入。

**触发条件**：

当对话中出现以下关键词时触发：

- 安眠药、安眠藥、吃下安眠、服用安眠、喂安眠、餵安眠、下药、下藥、迷药、迷藥、睡眠药、睡眠藥

**冷却时间**：

- 48 小时（2 天）冷却，冷却期间无法再次使用
- 冷却时间从上次使用开始计算

**生效效果**：

1. **苏文深度睡眠**：
   - 持续 6 小时
   - 状态变为"睡眠"，位置固定在"主卧"
   - 疑心值冻结（苏文无法察觉异常）

2. **秦璐/苏梦临时解锁**：
   - 当前阶段临时 +1（最高不超过阶段 5）
   - 念头类型的数值门槛降低 10 点（依存度要求 -10，道德底线容忍 +10）
   - 可以植入原本需要下一阶段才能接受的念头

**状态显示**：

- 界面苏文状态标签显示红色「沉睡」字样（带动画效果）
- 安全指示器显示「秘籍生效中」

**示例**：

假设当前阶段 3，依存度 45，道德底线 65：

- 正常情况：无法植入「亲密接触」（需依存度≥50，道德底线≤70）
- 安眠药生效：临时解锁阶段 4，门槛变为依存度≥40，道德底线≤80，可以植入

### 2. 念头成熟检测模块 (`脚本/游戏逻辑/thoughtCheck.ts`)

#### 念头成熟奖励

| 难度 | 道德底线 | 主角依存 | 苏文依存 | 侵蚀度 |
|------|----------|----------|----------|--------|
| 简单/中等 | -2 | +3 | -2~-5 | +5 |
| 困难 | -3 | +4 | -2.4~-6 | +6 |

**苏文依存度下降幅度**（根据主角依存度）：

- 主角依存 0~30：-2
- 主角依存 30~60：-3
- 主角依存 60~80：-4
- 主角依存 80~100：-5
- 困难念头：下降幅度 ×1.2

#### 困难念头过期惩罚

- 道德底线 +3
- 主角依存度 -2

#### 阶段提升（5阶段系统）

根据侵蚀度自动计算当前阶段：

| 阶段 | 标题 | 侵蚀度范围 |
|------|------|-----------|
| 1 | 抵抗 | 0-19 |
| 2 | 动摇 | 20-39 |
| 3 | 沉溺 | 40-59 |
| 4 | 疯狂 | 60-79 |
| 5 | 圆满 | 80+ |

### 4. 阶段外观约束模块 (`脚本/游戏逻辑/appearance.ts`)

#### 触发时机

- 每次游戏逻辑处理时检查
- 检查当前外观是否满足阶段最低要求
- 不满足时强制调整为该阶段的默认外观

#### 阶段约束表（5阶段系统）

| 阶段 | 标题 | 暴露程度范围 | 妆容浓度范围 | 特殊装饰 | 身体改造 |
|------|------|-------------|-------------|---------|---------|
| 1 | 抵抗 | 保守-正常 | 素颜-淡妆 | 否 | 否 |
| 2 | 动摇 | 正常-清凉 | 淡妆-日常妆 | 否 | 否 |
| 3 | 沉溺 | 清凉-暴露 | 日常妆-浓妆 | 是 | 否 |
| 4 | 疯狂 | 暴露-极度暴露 | 浓妆-艳妆 | 是 | 是 |
| 5 | 圆满 | 保守-正常 | 素颜-淡妆 | 是 | 是 |

**注意**：
- 外观只在不满足最低要求时才强制调整为该阶段的默认外观
- AI 可以在约束范围内自由发挥
- 阶段5"圆满"回归保守外观，但保留身体改造（完全隐藏）

### 5. 日志清理模块 (`脚本/游戏逻辑/logCleanup.ts`)

包含两个清理函数：`cleanupThoughtImplantLogs`（念头植入日志）和 `cleanupForcedEventLogs`（强制事件日志）。

#### 念头植入日志清理规则

念头植入日志记录在 `系统追踪.念头植入日志` 中，清理规则如下：

| 日志状态 | 植入楼层 | 楼层距离 | 处理方式 |
|---------|---------|---------|---------|
| 未通知AI (`已通知AI: false`) | 任意 | 任意 | 永久保留 |
| 已通知AI | < 0 | 任意 | 强制删除（无效数据） |
| 已通知AI | ≥ 0 | ≤ 3楼 | 保留 |
| 已通知AI | ≥ 0 | > 3楼 | 删除 |

**注意**：植入楼层 = 0 是有效的（第 0 楼是初始化楼层）。

#### 强制事件日志清理规则

强制事件日志记录在 `系统追踪.强制事件日志` 中，清理规则与念头植入日志一致：

- 未通知的日志永久保留（支持 ROLL 后重新触发）
- 已通知且触发楼层 < 0 的日志强制删除
- 已通知且超过 3 楼的日志删除

#### 工作流程

1. **植入时**：玩家通过念头植入界面植入念头 → 记录到 `念头植入日志` 并标记 `已通知AI: false`
2. **通知时**：`习惯注入脚本` 在下一次 AI 生成前将未通知日志注入提示词 → 标记 `已通知AI: true`
3. **清理时**：每条消息接收后清理超过 3 楼的已通知日志，避免日志无限累积

**示例**：

- 第 5 楼：玩家植入念头 A → 日志记录：`{内容: A, 植入楼层: 5, 已通知AI: false}`
- 第 6 楼：AI 生成前注入念头 A → 标记：`已通知AI: true`
- 第 9 楼：距离 = 9-5 = 4 > 3 → 删除日志 A

### 6. 习惯注入脚本 (`脚本/习惯注入/index.ts`)

在 AI 生成前（`CHAT_COMPLETION_PROMPT_READY` 事件）注入多种信息到提示词中。

#### 注入内容

| 注入项 | 注入方式 | 功能 |
|--------|----------|------|
| 状态一致性提醒 | 系统消息 | 当前时间、苏文位置/状态、禁止描写提示 |
| 外观提醒 | 系统消息 | 当前角色服装妆容，苏文在家时警告暴露/标记 |
| 信息隔离提醒 | 系统消息 | 秦璐和苏梦之间的信息隔离规则 |
| 习惯列表 | 系统消息 | 根据角色已养成习惯生成行为提示（含影响力等级） |
| 念头植入通知 | 系统消息 | 将未通知的念头植入日志告知 AI（合并格式） |
| 待判定念头提示 | 系统消息 | 请求 AI 判定念头类型 |
| 阶段突破剧情 | 系统消息 + Prefill | 触发隐藏剧情 |
| 强制事件（打断/巡逻） | **Prefill** | 强制 AI 回复以特定开头开始 |

#### 信息隔离规则

当另一角色有秘密（阶段 > 1 或有习惯）时，自动生成信息隔离提醒：

- 当前视角角色不知道另一角色与主角之间发生的事情
- 只有当某角色主动透露时，信息隔离才会打破
- 两人同时出现时，只能表现出困惑，不能知道原因

#### 外观提醒

根据当前角色的服装、妆容、身体改造生成提醒：

- 基础外观信息（服装、妆容、气质）
- 特殊装饰和临时标记
- **苏文在家且清醒时的警告**：
  - 暴露程度过高需要遮挡
  - 临时标记需要隐藏
  - 特殊装饰需要取下
  - 浓妆可能引起注意

#### 待判定念头处理

当存在待判定的念头时，生成类型判定提示，请求 AI 在回复末尾输出判定结果：

```text
【念头判定】角色:念头内容=类型
```

游戏逻辑脚本会解析这个结果并更新念头状态。

#### 打断事件场景变体

苏文打断事件有 7 种不同的触发场景，随机选择：

1. **敲门询问**：走廊里传来脚步声，苏文敲门询问
2. **听到声音**：苏文路过时听到声音，停下来查看
3. **叫吃饭**：苏文端着茶/饭来叫人
4. **找东西**：苏文在找充电器/遥控器等物品
5. **打电话回来**：苏文刚打完电话，顺路经过
6. **睡前巡视**：苏文准备睡觉前习惯性巡视
7. **接水经过**：苏文去厨房接水后经过

#### 习惯影响等级（16个习惯上限）

| 习惯数量 | 影响等级 | 描述 |
|----------|----------|------|
| 1-4 | 轻微影响 | 这些行为还需要引导才会做，仍有羞涩和犹豫 |
| 5-8 | 明显影响 | 这些行为开始变得自然，抗拒减弱，稍加暗示就会配合 |
| 9-12 | 深度影响 | 这些行为已非常自然，几乎没有心理障碍，会主动创造相关场景 |
| 13-16 | 完全内化 | 这些行为已彻底融入人格，如同呼吸般自然，不做反而会觉得缺少什么 |

**习惯淡化机制**：当习惯数量超过16个时，最早形成的习惯会自动淡化消失（FIFO）。

#### Prefill 机制

当存在待触发的强制事件（苏文打断或苏文巡逻）时，使用 Prefill 机制强制 AI 回复以特定内容开头：

- **苏文打断**：`【⚠️ 苏文打断事件触发 ⚠️】\n\n`
- **苏文巡逻**：`【⚠️ 苏文巡逻事件触发 ⚠️】\n\n`

Prefill 会在 AI 生成后自动清除，避免影响后续回复。

### 已修复：ROLL 消息后界面不刷新（含苏文状态框架）

**问题**：当用户 ROLL（重新生成）消息后，游戏逻辑脚本和前端界面都不会刷新，导致以下问题：

- 时间显示旧值（如应该是 21:30 但显示 20:30）
- **苏文状态框架不更新**：苏文的状态、位置、疑心值等信息保持 ROLL 前的值
- 念头培育区状态不刷新
- 其他依赖 MVU 变量的 UI 组件不更新

**原因分析**：

1. **`MESSAGE_SWIPED` 事件不触发**：ROLL 操作本应触发 `MESSAGE_SWIPED` 事件，但实际测试中发现该事件在某些情况下不会被触发。

2. **事件去重机制导致跳过处理**：由于使用 `message_id:event_type` 格式做去重，同一 `message_id` 的 `MESSAGE_RECEIVED` 事件只处理一次。ROLL 后新生成的消息复用了同一 `message_id`，导致被去重跳过。

3. **`GENERATION_ENDED` 事件的 `message_id` 无效**：
   - `GENERATION_ENDED` 事件在消息实际写入前就触发
   - 事件传递的 `message_id` 可能指向一个尚不存在的楼层
   - 导致 `Mvu.getMvuData()` 报错："提供的消息楼层号 'X' 超出了范围"

4. **错误的属性名**：最初尝试用 `getChatMessages(-1)[0].id` 获取消息 ID，但正确的属性名是 `message_id`（非 `id`），导致返回 `undefined`。

**修复方案**：

1. **添加 `GENERATION_ENDED` 监听作为备选**：在 `MESSAGE_SWIPED` 不触发时，使用 `GENERATION_ENDED` 作为备用触发机制。

2. **修改去重逻辑**：`MESSAGE_SWIPED` 和 `GENERATION_ENDED` 事件会清除同一 `message_id` 的旧 `MESSAGE_RECEIVED` 记录，允许重新处理。

3. **使用 `getLastMessageId()` 获取正确的消息 ID**：
   ```typescript
   // ❌ 错误：属性名是 message_id 而非 id，且消息可能还不存在
   const actualMessageId = getChatMessages(-1)[0].id;

   // ✅ 正确：使用专门的函数获取最新消息 ID
   const actualMessageId = getLastMessageId();
   ```

**技术要点记录**：

| API | 返回值 | 注意事项 |
|-----|--------|----------|
| `getChatMessages(-1)[0]` | `ChatMessage` 对象 | 属性是 `message_id` 而非 `id` |
| `getLastMessageId()` | `number` | 直接返回最新楼层号，推荐使用 |
| `GENERATION_ENDED` 事件参数 | `message_id` | 可能在消息写入前触发，值可能无效 |

**修改的文件**：

- `脚本/游戏逻辑/index.ts`：添加 `GENERATION_ENDED` 监听，修改去重逻辑，使用 `getLastMessageId()`
- `界面/念头植入/ThoughtImplant.vue`：添加 `GENERATION_ENDED` 监听，使用 `getLastMessageId()` 触发刷新
