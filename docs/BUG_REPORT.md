## BUG #1: 真好结局与完美真爱结局系统文档与实际不一致

**严重程度**: 中等

**问题描述**:
根据 `docs/真好结局系统.md` 文档，真好结局系统应该处理两种结局（真好结局和完美真爱结局），共用10阶段剧情流程。但实际上代码中存在两个独立的文件：

## BUG #5: 混乱结局系统容错次数文档与代码实现差异

**严重程度**: 中等

**问题描述**:

```
| 前置条件 | 容错次数 | 说明 |
|---------|---------|------|
| 未完成场景3 | 2次 | 第1次警告，第2次标记 |
| 已完成场景3 | 3次 | 有记忆基础，多1次容错 |
```

但 `confusionEndingSystem.ts` 实际实现使用的是**混乱度数值系统**:
```typescript
// 进入场景5时混乱度固定为80
// - 完美路线（连贯性=3）：每次+7，3次到100（80→87→94→101）
// - 非完美路线（连贯性<3）：每次+10，2次到100（80→90→100）
```

**差异说明**:
- 文档描述的是"未完成场景3" vs "已完成场景3"
- 代码实现的是"完美路线（连贯性=3）" vs "非完美路线（连贯性<3）"

这两个条件并不等价:
- 完成场景3不一定意味着记忆连贯性=3
- 记忆连贯性=3需要按顺序完成场景1→2→3

**建议**: 更新文档以准确反映代码实现的混乱度数值机制。

**✅ 确认：代码逻辑正确，文档需更新 (2026-01-18)**

   - 记忆连贯性=3 意味着玩家按顺序完成场景1→2→3且选择正确

   | 完美路线 | 连贯性=3 | +7/次 | 3次（80→87→94→101） |
   | 非完美路线 | 连贯性<3 | +10/次 | 2次（80→90→100） |

## BUG #13: 场景5记忆连贯性锁定时机错误

**严重程度**: 严重

**问题描述**:
当前代码在**首次进入场景5时**就锁定记忆连贯性，但根据设计意图，应该在**完成场景5时**（完成度=100%（12步全部完成））才锁定。

**当前代码** (`scene5System.ts:362-367`):
```typescript
export function lockScene5EntryCoherence(data: SchemaType): void {
  const coherence = calculateMemoryCoherence(data);
  data.梦境数据.记忆连贯性 = coherence.level;
  data.梦境数据.场景5进入时连贯性 = coherence.level;
}
```
此函数在首次进入场景5时调用（`index.ts:880`），导致连贯性在进入时就被锁定。

**设计意图**:

场景5需要进入2次才能完成（单次最多10步，需要12步完成）：

- 第1次进入：最多完成步骤0-9（10步），完成度约50%-83%，**未完成**，不应锁定连贯性
- 第2次进入：完成剩余步骤10-11，达到100%完成度，**完成时锁定连贯性**

**时间计算**:

| 时间 | 状态 |
|------|------|
| 08:00 | 进入引导开始 |
| 09:00 | 步骤0（第1步剧情） |
| 10:00-18:00 | 步骤1-9 |
| 19:00 | 退出引导开始 |
| 20:00 | 退出完成 |

单次进入实际可完成：步骤0-9（共10步），无法完成全部12步。

**完美真爱结局路线设计**:

玩家可以在 Day 1、Day 2 或 Day 3 的白天进入场景5第1次，都不影响完美真爱结局的达成。

**路线A（Day 1 进入场景5第1次）**:

| 天数 | 白天 | 晚上 |
|------|------|------|
| Day 1 | 场景5 第1次（11:00进入，7-8步，未完成，**不锁定**） | 场景1 |
| Day 2 | - | 场景2 |
| Day 3 | - | 场景3 |
| Day 4 | 场景5 第2次（08:00进入，完成，**锁定连贯性=3**）→ 20:00退出 | 场景4 |
| Day 5 | 11:00 完美真爱结局引导开始 | 00:00 结局判定 |

**路线B（Day 2 进入场景5第1次）**:

| 天数 | 白天 | 晚上 |
|------|------|------|
| Day 1 | - | 场景1 |
| Day 2 | 场景5 第1次（11:00进入，7-8步，未完成，**不锁定**） | 场景2 |
| Day 3 | - | 场景3 |
| Day 4 | 场景5 第2次（08:00进入，完成，**锁定连贯性=3**）→ 20:00退出 | 场景4 |
| Day 5 | 11:00 完美真爱结局引导开始 | 00:00 结局判定 |

**路线C（Day 3 进入场景5第1次）**:

| 天数 | 白天 | 晚上 |
|------|------|------|
| Day 1 | - | 场景1 |
| Day 2 | - | 场景2 |
| Day 3 | 场景5 第1次（11:00进入，7-8步，未完成，**不锁定**） | 场景3 |
| Day 4 | 场景5 第2次（08:00进入，完成，**锁定连贯性=3**）→ 20:00退出 | 场景4 |
| Day 5 | 11:00 完美真爱结局引导开始 | 00:00 结局判定 |

**关键条件**:

1. 场景5**完成时**（不是进入时），场景1-2-3必须都已完成
2. 5个场景全部完成且正确选择
3. 场景5完成度=100%（12步全部完成）时锁定连贯性

**设计优势**:

- 给玩家提供多条隐藏路线，增加游戏探索性
- 玩家可以"试探性"进入场景5，不会立即锁定连贯性
- 只有在真正完成场景5时才决定最终结局

**影响**:
- 当前实现会在Day 1首次进入时锁定连贯性=0
- 导致玩家无法通过"先试探性进入场景5，再完成场景1-2-3"的路线达成完美真爱结局
- 完美真爱结局变成不可能达成的隐藏结局

**修复建议**:

## 设计补充：场景5"试探性进入"机制

### 多周目体验差异

场景5的设计让多次游玩的玩家有完全不同的体验：

**首次游玩（不知道隐藏路线）**：
- 玩家按顺序 Day 1→2→3→4 完成场景1-4
- Day 5 白天进入场景5，但时间不够完成2次
- 无法达成完美真爱结局（只能达成真好结局或假好结局）

**二周目（发现隐藏路线）**：
- 玩家在 Day 1/2/3 白天"试探性"进入场景5（未完成）
- 继续完成场景1-2-3
- Day 4 完成场景5（锁定连贯性=3）+ 场景4
- Day 5 触发完美真爱结局

### 场景5完成时机决定结局

| 完成场景5的时机 | 连贯性 | 可能的结局 |
|----------------|--------|------------|
| Day 1-3 完成2次场景5（未完成1-2-3） | 0-2 | 真好结局/假好结局 |
| Day 4 完成场景5（已完成1-2-3） | 3 | **完美真爱结局** |
| Day 5 完成场景5（时间不足） | - | 无法完成，普通结局 |

### "试探性进入"机制设计

**核心概念**：场景5未完成时（<100%，即未完成全部12步），视为"试探性进入"，不产生负面后果。

**需要优化的内容**：

1. **记忆连贯性**：完成时才锁定，未完成不锁定
2. **记忆混乱度**：未完成退出时回滚到进入前的值
3. **混乱结局标记**：未完成退出时清除标记
4. **步骤进度**：保留，下次进入可继续

**流程图**：
```
场景5进入
    ↓
[游玩中] ← 暂存进入前状态，不锁定连贯性
    ↓
退出时检查完成度
    ↓
┌─────────────────┬─────────────────┐
│  完成度 < 100%   │  完成度 ≥ 80%   │
│  (试探性进入)    │  (正式完成)      │
├─────────────────┼─────────────────┤
│ - 不锁定连贯性   │ - 锁定连贯性     │
│ - 回滚混乱度     │ - 结算混乱度     │
│ - 清除混乱标记   │ - 检测混乱结局   │
│ - 不计入已完成   │ - 加入已完成列表 │
│ - 保留步骤进度   │ - 标记场景完成   │
└─────────────────┴─────────────────┘
```

**新增数据字段**：
```typescript
场景5: {
  // 现有字段...
  进入前混乱度: z.number().optional(),     // 进入前的混乱度，未完成时回滚
  进入前混乱标记: z.object({...}).optional(), // 进入前的混乱结局标记
}
```

**设计意义**：
- 完美真爱结局成为真正的"隐藏结局"
- 需要玩家二周目才能发现路线
- 增加游戏重玩价值和探索乐趣

## BUG #30: 场景5越轨检测使用未锁定的连贯性字段

**严重程度**: 中等

**问题描述**:
在场景5中检测性行为时，`isPerfectMemoryRoute` 函数直接读取 `data.梦境数据.记忆连贯性` 字段来判断是完美路线还是非完美路线。但根据 Bug #13 的修复，连贯性字段是在场景5**完成时**（100%完成度）才锁定的，不是进入时。

**设计文档说明**（`混乱度数值设计.md`）:
- 完美路线（连贯性=3）：每次越轨+7，3次到达100
- 非完美路线（连贯性<3）：每次越轨+10，2次到达100

**问题场景**:
1. 玩家按顺序完成场景1→2→3（应该是完美路线，连贯性=3）
2. 进入场景5，此时 `记忆连贯性` 字段还是默认值 0
3. 玩家在场景5中触发性行为
4. `isPerfectMemoryRoute` 读取 `记忆连贯性=0`，判定为非完美路线
5. 增量使用 +10 而不是正确的 +7
6. 玩家只有2次容错机会，而应该有3次

**根本原因**:

```typescript
// 原代码 - 读取可能未锁定的字段
export function isPerfectMemoryRoute(data: SchemaType): boolean {
  return (data.梦境数据.记忆连贯性 ?? 0) === 3;
}
```

**✅ 已修复 (2026-01-18)**:

**修复内容**:

使用 `calculateMemoryCoherence()` 函数实时计算连贯性，而不是读取可能未锁定的字段：

```typescript
import { calculateMemoryCoherence } from './scene5System';

export function isPerfectMemoryRoute(data: SchemaType): boolean {
  // Bug #30 修复：使用实时计算的连贯性
  const coherence = calculateMemoryCoherence(data);
  return coherence.level === 3;
}
```

**修复效果**:

| 场景 | 修复前 | 修复后 |
|------|--------|--------|
| 完成1-2-3后进入场景5 | 判定为非完美（+10，2次触发） | 判定为完美（+7，3次触发） |
| 未完成1-2-3进入场景5 | 判定为非完美（+10，2次触发） | 判定为非完美（+10，2次触发） |

- `index.ts:1265` - `validateAndRestoreData(data)`
- `dataProtection.ts:75` - `'梦境数据.记忆混乱度': ProtectionLevel.SCRIPT_ONLY`

**✅ 已修复 (2026-01-18)**:

## 优化 #1: 结局系统世界书拆分

**类型**: 优化

**日期**: 2026-01-18

**问题描述**:
原来的结局系统设计中，所有6种结局的详细内容（完美真爱、真好、假好、坏、混乱、普通）都放在单一的 `结局系统.yaml` 文件中。一旦玩家进入梦境（`已进入过梦境: true`），全部约270行的结局内容都会发送给AI。

这存在以下问题：
1. **浪费token**：触发完美真爱结局时，AI不需要知道坏结局、混乱结局的描写指南
2. **可能干扰AI**：其他结局的描写风格可能影响当前结局的AI输出
3. **后日谈内容混杂**：三种好结局的后日谈剧情也全部发送，即使只触发了其中一种

**优化方案**:

将结局系统拆分为多个独立条目，按条件激活：

### 文件结构

| 文件 | 用途 |
|------|------|
| `结局系统概述.yaml` | 通用判定逻辑（约30行） |
| `完美真爱结局.yaml` | 完美真爱结局描写指南 |
| `真好结局.yaml` | 真好结局描写指南 |

## BUG #39: 结局世界书条目无法通过正则匹配变量列表激活

**严重程度**: 严重

**日期**: 2026-01-18

**问题描述**:

在优化 #1 中，我们将结局系统拆分为多个独立条目，使用正则表达式匹配变量列表中的结局状态来激活对应条目。但这个设计存在根本性问题：**正则无法匹配到变量列表的内容**。

**设计的正则**:
```yaml
# 假好结局条目
关键字: /当前结局["']?:\s*["']?假好结局/
扫描深度: 100
```

**预期行为**:
- 正则匹配变量列表中的 `当前结局: 假好结局`
- 假好结局条目被激活

**实际行为**:
- 正则**永远无法匹配**
- 假好结局条目不会被激活

**根本原因**:

1. **扫描深度的误解**：扫描深度扫描的是**聊天正文**（玩家消息+AI回复），不是深度插入的系统消息

2. **变量列表的插入位置**：
   ```yaml
   插入位置:
     类型: 指定深度
     角色: 系统
     深度: 0
   ```
   变量列表被插入到深度0的系统消息，不在聊天正文中

3. **递归机制被阻断**：变量列表设置了 `不可激活其他条目: true`
   ```yaml
   递归:
     不可被其他条目激活: true
     不可激活其他条目: true  # <-- 问题所在
   ```

   根据酒馆的递归机制：
   - 条目被激活后，其内容会加入"欲扫描文本"
   - 其他绿灯条目可以在递归中匹配这些新加入的内容
   - 但 `不可激活其他条目: true` 阻止了变量列表的内容加入欲扫描文本
   - 因此正则**永远无法匹配到变量列表中的内容**

**影响**:
- 结局系统拆分后的所有条目（完美真爱结局、真好结局、假好结局等）都无法被激活
- 后日谈条目也无法被激活
- AI 收不到结局相关的描写指南

**解决方案**:

### 方案A：移除递归限制（有风险）

修改变量列表的递归设置：
```yaml
递归:
  不可被其他条目激活: true
  # 不可激活其他条目: true  # 删除这行
```

**风险**：可能导致其他意外的递归激活，需要仔细测试

### 方案B：使用脚本 injectPrompts 注入（推荐）

既然结局系统的阶段引导已经通过脚本注入（`falseEndingSystem.ts` 等），结局的描写指南也应该通过脚本注入：

```typescript
// 在 promptInjection.ts 中
if (data.结局数据.当前结局 === '假好结局') {
  injectPrompts([{
    content: 假好结局描写指南,
    position: 'before_char',
    // ...
  }]);
}
```

**优点**：
- 不需要修改变量列表的递归设置
- 完全由脚本控制，逻辑清晰
- 避免递归机制的复杂性

### 方案C：创建激活专用条目

创建一个蓝灯条目，始终输出结局激活标记：
```yaml
- 名称: 结局激活标记
  启用: true
  激活策略:
    类型: 蓝灯
  内容: |
    <%_ if (getvar('结局数据.当前结局') === '假好结局') { _%>
    【激活标记：假好结局】
    <%_ } _%>
```

然后让结局条目匹配这个标记。

**涉及文件**:
- `src/赵霞/世界书/index.yaml` - 结局条目配置（当前设计无效）
- `src/赵霞/世界书/变量/变量列表.yaml` - 递归设置
- `src/赵霞/脚本/游戏逻辑/promptInjection.ts` - 可能需要添加结局描写指南注入

```typescript
  // ...

```

```typescript
function getDreamSceneGuidance(data: SchemaType): string | null {
```

## BUG #29: 混乱结局触发后长时间处于精神崩溃状态

**日期**: 2026-01-18

**严重程度**: 中等

**问题描述**:
在场景5中触发混乱结局后，玩家会长时间处于"精神崩溃"状态。原设计为标记混乱结局后延迟1小时触发，但场景5是日间梦境（08:00-20:00），需要完成12步骤才会退出，导致玩家在触发混乱结局后仍需继续在场景5中等待，体验不好。

**原因分析**:
- `markConfusionEnding()` 设置 `触发时间 = 当前时间 + 1小时`
- `checkConfusionEnding()` 检测 `当前时间 >= 触发时间` 才返回结局模板
- 场景5内即使混乱结局已标记，游戏阶段仍为"梦境"，玩家被困在场景5中

**修复方案**:
1. **立即触发**: 将 `markConfusionEnding()` 中的触发时间从 `currentGlobalHour + 1` 改为 `currentGlobalHour`（立即触发）
2. **强制退出**: 在 `applyConfusionEndingState()` 中添加逻辑，当游戏阶段为"梦境"时强制切换到"日常"阶段

```typescript
// markConfusionEnding() 修改
data.梦境数据.混乱结局.触发时间 = currentGlobalHour; // 立即触发，不再延迟

// applyConfusionEndingState() 新增
if (data.世界.游戏阶段 === '梦境') {
  data.世界.游戏阶段 = '日常';
  data.世界.状态栏需要刷新 = true;
  console.info(`[混乱结局] 强制退出梦境阶段`);
}
```

**修复文件**: `src/赵霞/脚本/游戏逻辑/confusionEndingSystem.ts`

**状态**: ✅ 已修复
