# 秦璐重置版 — 念头判定机制（已确认方案）

> 状态：方向已敲定，细节待续。
> 记录时间：2026-06-22

---

## 一、判定方式：全交给 AI，砍掉关键词库

- 念头分类**全部交给 AI 判定**（方案 B），**砍掉旧版的本地关键词匹配库**（300+ 词条 + 两套并存的关键词表）。
- 理由：
  - AI 语义判定本来就比关键词准，能戳穿委婉语/文字游戏。
  - 砍掉词库 = 少维护、少误判源、更符合「数值/系统精简」铁律。
- 代价：每个念头要等 AI 判一轮（约 1 轮延迟），可接受。

---

## 二、核心问题：旧判定协议为什么脆弱

旧秦璐的判定链路：
1. 让 AI 在正文最后一行输出 `【念头判定】角色:念头内容=类型`
2. 脚本用正则去正文里抠这行
3. 靠「念头内容字符串相等」匹配是哪条念头

脆弱点：
- AI 没输出 / 输出到正文中间 / 用了全角符号 → 正则失败
- AI 复述念头内容时改一个字、加引号 → 内容比对失配
- 判定行出现在正文里被玩家看到 → 破坏沉浸
- 失败只能等 2 小时超时 → 强制降级成「陪伴交流」，把玩家精心种的极端念头**静默洗白**
- 每轮重复注入完整 10 大类说明 → 浪费 token

## 三、解决方案：念头 ID 化 + 走 MVU 变量判定

> 借鉴云霜凝核心思想：**永远不让 AI 用「复述内容」来定位条目，而是用「稳定的 key/id」。**
> 云霜凝的道具/性癖/场景都用 `Record<key, value>`（带 key 的字典），脚本永远按 key 直接取值，从不靠内容反查。

### 3.1 数据结构：念头存成「带 ID 的字典」，不是数组

```
念头列表: Record<id, {
  内容: string,          // 玩家输入的原文，前端写入，AI 不许改
  类型: 枚举 | '待判定',  // 唯一让 AI 写的字段
  开发进度: number,
  ...（培育/状态等字段，用枚举状态机，不用魔法数字）
}>
```

示例：
```
{
  "念头_1719060000000":      { 内容:"想和他多聊聊", 类型:"待判定" },
  "念头_1719060005000_376":  { 内容:"想靠在他肩上", 类型:"待判定" }
}
```

### 3.2 ID 怎么生成（关键澄清）

- **ID 由前端在玩家点「植入」那一刻用 JS 代码生成**，不是 AI 生成，也不是 MVU 生成。AI/MVU 只「使用」这个已存在的 ID，不「创造」它。
- 生成方式：时间戳（+ 可选随机后缀）

```js
// 基础版（毫秒时间戳，玩家不可能同毫秒植入两个，天然唯一）
const id = '念头_' + Date.now();
// → 念头_1719060000000

// 保险版（加随机后缀，绝对不撞车）
const id = '念头_' + Date.now() + '_' + Math.floor(Math.random() * 1000);
// → 念头_1719060000000_376
```

- 那个 `abc`/`376` 只是随机后缀，让 ID 绝对唯一。
- 对标云霜凝：身份标识由系统侧预先确定（道具名当 key、留影石用计数器），AI 只引用不创造。

### 3.3 植入那一刻前端做三件事

> 前端输入限制：念头内容最多 **10 个字**，鼓励玩家输入短促、明确的念头种子。

```js
// 1. 生成唯一 ID
const id = '念头_' + Date.now();

// 2. 用 ID 作为 key 写进 MVU 变量（内容=玩家输入，类型先填"待判定"）
念头列表[id] = { 内容: 玩家输入文字, 类型: '待判定', 开发进度: 0 };

// 3. 落盘到 MVU
saveToMvu(念头列表);
```

### 3.4 判定流程

1. 前端植入 → 分配 ID，类型先填「待判定」。
2. 植入后 3 楼内动态注入提示给 AI：**「`念头_xxx` 的内容是『…』，请判定它的类型，写入 `念头列表.念头_xxx.类型`」**——AI 引用的是 **ID**，不复述内容。
3. AI 通过 **MVU 变量命令**往 `念头列表.念头_xxx.类型` 写一个**枚举值**（不在正文里写判定行）。
4. 脚本在 `VARIABLE_UPDATE_ENDED` 直接读 `念头列表["念头_xxx"].类型`，O(1) 取值，不用正则、不用内容比对。

### 3.5 为什么这就不脆弱了

| 旧问题 | 新方案如何解决 |
|--------|--------------|
| AI 改一个字就失配 | AI 引用 ID 不复述内容，改字无影响 |
| 正则抠正文易失败/污染正文 | 走 MVU 变量，不写正文行 |
| 解析失败只能超时降级洗白 | AI 漏判 → 类型仍是「待判定」→ 下一轮自然重试，不洗白 |
| AI 写错类型会出脏值 | 类型是 `z.enum(...)`，Zod 自动把非法值落回「待判定」，不崩不脏 |
| 数组删条目下标漂移 | Record 用 key 寻址，增删不影响其它条目 |

## 三bis、植入体验：乐观植入 + 判定中占位（已确认）

### 问题：AI 判定有一轮延迟

"念头类型全交 AI" → AI 要等下一轮回复才判出类型。玩家点「植入」当下**还不知道能不能植入**，会"提交后悬空"。

### 关键拆解：植入要判两件事，时机不同

| 判什么 | 谁判 | 时机 |
|--------|------|------|
| ① 念头是什么**类型** | AI | 等下一轮（语义判定，求准） |
| ② 该类型**够不够阶段/数值** | 脚本 | 一旦类型已知，**纯数值比对，瞬时** |

> 即：延迟只在 ①。一旦 AI 判出类型，② 立刻出结果。

### 解决：乐观植入 + 判定中占位（已确认）

1. **玩家点植入 → 立即收下**，分配 ID，状态 = `判定中`，**当场进培育区占位显示**（发光/"判定中"标记），不卡手。
   - 把这一轮延迟**包装成设定**："念头需要在下一次互动中生根/验证"，而非 bug。
2. **下一轮 AI 判出类型** → 脚本立即做 ② 阶段/数值校验：
   - **合格** → 状态转 `培育中`，开始按楼层推进进度。
   - **不合格**（超阶段 / 数值不够，且不在心防松动窗口/道具解锁期）→ 状态转 `未达标`，**标红提示**"她现在还接受不了，需要先推进关系"。
3. **不合格念头的去向（已确认：标红 + 退回/保留）**：
   - 明确反馈，**不静默洗白、不暗中扣数值**。
   - 玩家可**退回**（删除，了结）或**保留**（留着，等阶段/数值够了或开窗时再处理）。

### 念头状态机（完整生命周期）

```
判定中 ──AI判出类型──→ ┬─ 合格 ─→ 培育中 ──进度满──→ 已成熟（转习惯，结算数值）
                      └─ 不合格 ─→ 未达标 ──玩家选择──→ 退回(删除) / 保留(待条件满足或开窗)
                                                  保留中若开窗或达标 ─→ 培育中
（培育中/未达标 超过保留楼数上限可选 ─→ 已过期）
```

- 状态用**显式枚举**（判定中/培育中/未达标/已成熟/已过期），不用魔法数字。
- `判定中`：AI 漏判则保持，下一轮重试，不降级（对接§五漏判容错）。

## 四、顺手解决的旧问题

- **枚举状态机取代魔法数字**：念头状态用显式字段（判定中/培育中/未达标/已成熟/已过期，见三bis 状态机），不再用「需要时间=0 表示待判定、=26 表示软拒绝」这种暗号。
- **判定说明只在植入后 3 楼内动态注入**：待判定念头在植入后 3 楼内向 AI 提示 10 大类与写入路径；不再全局常驻重复注入 → 省 token，对齐数据隔离铁律。
- **砍掉关键词库**：少维护 300+ 词条，少一套误判源。

---

## 五、健壮性（直接照搬云霜凝）

- **内容由前端写、脚本保护**：AI 只许写「类型」这一个判定字段，不许改「内容」（对标云霜凝服装/道具回滚保护）。
- **枚举 + 兜底**：类型用枚举，非法值落回「待判定」。
- **漏判容错**：AI 没判 → 保持「待判定」重试，不报错、不降级。
- **ID 做幂等键**：同一念头若被多轮处理，用 ID 去重防重复。

---

## 六、越级闸门：心防松动窗口

- 念头植入判定在校验「阶段/数值门槛」前，先检查角色是否处于**心防松动窗口**（楼层周期触发）。窗口内：有效阶段 +1、免数值门槛，放行高一级念头。
- 详见《心防松动窗口.md》。本机制与判定通道共用同一套阶段/门槛校验，越级闸门只是在校验前抬高有效阶段并跳过门槛。
